## Обзор архитектуры  
  
### Frontend (Emerald-Grove-Frontend)  
- **Технологии**: TypeScript, React, IndexedDB  
- **Хранилище**: IndexedDB (chrome.storage.local для auth)  
- **Ключевые компоненты**:  
    - `background/` - Service Worker для синхронизации  
    - `content/` - Content script для извлечения данных со страниц  
    - `library/` - Основной UI библиотеки  
    - `popup/` - Popup UI для расширения  
    - `shared/` - Общие типы, API, репозиторий  
  
### Backend (Emerald-Grove-Backend)  
- **Технологии**: Java, Spring Boot, JPA, PostgreSQL  
- **Хранилище**: PostgreSQL  
- **Ключевые компоненты**:  
    - `controller/` - REST API контроллеры  
    - `service/` - Бизнес-логика  
    - `entity/` - JPA сущности  
    - `repository/` - JPA репозитории  
    - `dto/` - Data Transfer Objects

## КРИТИЧЕСКИЕ ПРОБЛЕМЫ ДЛЯ МИГРАЦИИ

### 1. Отсутствие полей isFavorite и isReadLater в Backend

**Проблема:**
Frontend использует поля `isFavorite` и `isReadLater` в типе Article:
```typescript
interface Article {
  isFavorite?: boolean
  isReadLater?: boolean
}
```

Backend entity `Article` не содержит этих полей:
```java
@Entity
public class Article extends BaseEntity {
    // id, externalId, user, title, url, description, aiStatus, notes
    // НЕТ isFavorite и isReadLater
}
```

**Последствия:**
- Флаги хранятся только локально в IndexedDB
- При sync на другом устройстве эти флаги теряются
- System sections (FAVORITES, READ_LATER) не работают корректно после sync
- Нельзя реализовать backend-фильтрацию по этим флагам

**Решение:**
1. Добавить поля `isFavorite` и `isReadLater` в entity `Article`
2. Обновить DTO: `SyncArticleRequestDto`, `ArticleSyncDto`, `SyncArticlePayloadResponseDto`
3. Обновить логику sync в `ArticleServiceImpl`
4. Добавить endpoints для управления флагами (или включить в существующий sync)
5. Добавить фильтрацию в GET /api/articles по этим флагам

---

### 2. Отсутствие bulk sync для статей

**Проблема:**
Для коллекций реализован bulk sync:
- POST /api/collections/sync
- POST /api/collections/links/sync

Для статей существует только единичный sync:
- POST /api/articles/sync (одна статья)

**Последствия:**
- При оффлайн-создании множества статей нельзя отправить их одной операцией
- Нельзя эффективно синхронизировать состояние после длительного оффлайна
- Несоответствие с паттерном, использованным для коллекций

**Решение:**
1. Создать POST /api/articles/sync/bulk для массового sync статей
2. Использовать аналогичную логику как в коллекциях:
   - Если статья с externalId существует - обновить
   - Если не существует - создать
   - Silent skip для отсутствующих referenced сущностей
3. Добавить bulk sync для заметок (если нужно отдельно от статей)
4. После bulk sync обязательно делать повторный pull server state

---

### 3. Отсутствие bulk sync для удалений

**Проблема:**
Backend не имеет механизма для массовой синхронизации удалений:
- Нет endpoint для bulk delete статей
- Нет endpoint для bulk delete коллекций
- Нет endpoint для bulk delete links

**Последствия:**
- Если пользователь удалил 10 статей оффлайн, при sync они не удалятся на backend
- При sync на другом устройстве удаленные статьи появятся снова
- Несоответствие локального и серверного состояния

**Решение:**
1. Создать POST /api/articles/sync/deletions с массивом externalId для удаления
2. Создать POST /api/collections/sync/deletions
3. Создать POST /api/collections/links/sync/deletions
4. Frontend должен вести отдельную очередь удаленных externalId
5. После sync deletions делать повторный pull server state

---

### 4. Отсутствие conflict resolution на основе временных меток

**Проблема:**
Backend не использует клиентские временные метки для разрешения конфликтов:
- Коллекция с тем же externalId просто обновит name
- Link с той же парой articleExternalId + collectionExternalId не создастся дубликат
- Нет логики "последняя запись побеждает" на основе updatedAt

**Последствия:**
- Если пользователь изменил статью на двух устройствах оффлайн, изменения могут потеряться
- Нет предсказуемого поведения при конфликтах
- Нельзя реализовать полноценную offline-first синхронизацию

**Решение:**
1. Добавить поле `clientUpdatedAt` в DTO для sync
2. В sync логике сравнивать clientUpdatedAt с текущим серверным updatedAt
3. Если clientUpdatedAt > серверный updatedAt - принять клиентские изменения
4. Если clientUpdatedAt < серверный updatedAt - отклонить или merge
5. Рассмотреть использование vector clocks или CRDT для сложных случаев

### 5. Несоответствие между frontend и backend структурами данных

**Проблема:**
Frontend Article содержит поля, которых нет в backend:
```typescript
{
  syncStatus?: 'SYNCED' | 'PENDING_SYNC'  // Только frontend
  isFavorite?: boolean                      // Нет в backend
  isReadLater?: boolean                     // Нет в backend
}
```

Backend ArticleSyncDto не содержит этих полей:
```java
public record ArticleSyncDto(
    Long articleId,
    String externalId,
    String url,
    String title,
    String description,
    Long createdAt,
    Long updatedAt,
    String aiStatus,
    List<ArticleNoteDto> notes
) {}
```

**Последствия:**
- При sync эти данные теряются
- Frontend вынужден хранить их отдельно и восстанавливать после sync
- Сложность в поддержании консистентности

**Решение:**
1. Добавить недостающие поля в backend entity и DTO
2. Либо документировать, что эти поля остаются локальными и не sync-ятся
3. Если второе - явно отделить локальные поля от sync-ируемых в типах

### 6. Отсутствие поддержки System Sections в Backend

**Проблема:**
Frontend имеет SystemSection тип:
```typescript
type SystemSection = 'ALL' | 'RECENT' | 'FAVORITES' | 'READ_LATER'
```

Backend GET /api/articles возвращает все статьи без фильтрации:
```java
public List<ArticleSyncDto> getAll(Long userId) {
    return articleRepository.findAllByUserId(userId)...
}
```

**Последствия:**
- Фильтрация FAVORITES и READ_LATER работает только локально в IndexedDB
- При sync на новом устройстве эти секции будут пустыми или некорректными
- Нет возможности получить "RECENT" с сервера

**Решение:**
1. Добавить query параметры в GET /api/articles:
   - ?filter=favorites
   - ?filter=readLater
   - ?filter=recent&days=7
2. Реализовать фильтрацию в repository
3. Frontend должен использовать эти endpoints вместо локальной фильтрации после sync

---

### 7. Silent skip для отсутствующих данных в sync

**Проблема:**
В коллекциях backend молчаливо пропускает link'и, если статья или коллекция не найдены:
- Backend возвращает 200 OK
- Конкретный link не создается
- Frontend не знает, какие именно link'ы были пропущены

**Последствия:**
- Frontend не может корректно обработать ошибку
- Пользователь не видит, что часть данных не синхронизировалась
- Потеря данных без уведомления

**Решение:**
1. В ответ sync endpoint возвращать список:
   - успешно созданных/обновленных сущностей
   - пропущенных сущностей с причиной (article not found, collection not found)
2. Frontend должен анализировать ответ и:
   - Повторно sync пропущенные сущности после pull
   - Показать пользователю предупреждение о несинхронизированных данных
3. Рассмотреть возврат 207 Multi-Status для частичного успеха

### 8. AI Status синхронизация

**Проблема:**
Frontend имеет тип AIStatus:
```typescript
type AIStatus = 'PENDING' | 'DONE' | 'FAILED'
```

Backend хранит aiStatus в entity Article и имеет endpoints для AI:
- GET /api/articles/{externalId}/ai
- POST /api/articles/{externalId}/ai/retry

**Вопросы:**
- Корректно ли sync-ится aiStatus при sync статьи?
- Что происходит, если AI анализ запускается на одном устройстве, а пользователь смотрит на другом?
- Нужно ли отдельное поле для client-side AI status?

**Решение:**
1. Уточнить требования к AI синхронизации
2. Если AI статус должен sync-иться - включить в ArticleSyncDto
3. Рассмотреть отдельный endpoint для pull AI статусов
4. Добавить polling или WebSocket для real-time обновлений AI статуса

## ПАТТЕРНЫ ИЗ ЭТАПА 1, КОТОРЫЕ НУЖНО ПРИМЕНЯТЬ

### 1. Backend как источник истины после sync
- IndexedDB используется как кэш, оффлайн-слой, временное хранилище
- После успешного sync актуальное состояние подтягивается с backend
- Frontend должен делать повторный pull после любого bulk sync

### 2. Разделение online CRUD и offline bulk sync
- Online: прямые REST вызовы (POST, PUT, DELETE)
- Offline: накопление изменений в pending queue, затем bulk sync
- Нельзя смешивать эти подходы для одной сущности

### 3. Отдельная синхронизация связанных сущностей
- Коллекции и membership-связи sync-ятся отдельно
- Аналогично: статьи и заметки могут sync-иться отдельно
- Frontend должен sync-ить все связанные сущности для консистентности

### 4. Использование externalId как основного идентификатора
- Локальный id должен совпадать с backend externalId
- Для online-created сущностей externalId берется из ответа backend
- Для offline-created сущностей externalId генерируется локально

### 5. Silent skip требует повторного pull
- Если backend молчаливо пропускает данные, frontend не может полагаться на HTTP код
- Обязательно делать повторный pull server state после sync
- Либо изменить backend на возвращение детальной информации о пропущенных данных
