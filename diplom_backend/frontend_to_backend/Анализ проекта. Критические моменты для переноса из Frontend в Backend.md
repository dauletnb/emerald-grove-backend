## Обзор архитектуры  
  
### Frontend (Emerald-Grove-Frontend)  
- **Технологии**: TypeScript, React, IndexedDB  
- **Хранилище**: IndexedDB (chrome.storage.local для auth)  
- **Ключевые компоненты**:  
    - `background/` - Service Worker для синхронизации  
    - `content/` - Content script для извлечения данных со страниц  
    - `library/` - Основной UI библиотеки  
    - `popup/` - Popup UI для расширения  
    - `shared/` - Общие типы, API, репозиторий  
  
### Backend (Emerald-Grove-Backend)  
- **Технологии**: Java, Spring Boot, JPA, PostgreSQL  
- **Хранилище**: PostgreSQL  
- **Ключевые компоненты**:  
    - `controller/` - REST API контроллеры  
    - `service/` - Бизнес-логика  
    - `entity/` - JPA сущности  
    - `repository/` - JPA репозитории  
    - `dto/` - Data Transfer Objects

## РИСКИ И ЗАМЕЧАНИЯ  
  
1. **Синхронизация данных**: Необходимо реализовать корректную синхронизацию между IndexedDB и backend для всех новых функций  
2. **Миграция существующих данных**: Пользователи с существующими локальными данными потеряют их при первом sync после добавления backend функционала  
3. **Производительность**: Поиск и фильтрация на backend может быть медленнее для небольших наборов данных по сравнению с клиентской фильтрацией  
4. **Оффлайн режим**: Коллекции и флаги должны продолжать работать оффлайн с последующей синхронизацией  
5. **Консистентность**: Убедиться, что frontend и backend используют одинаковую логику валидации и бизнес-правила

## ЗАКЛЮЧЕНИЕ  
  
Наиболее критичными для переноса являются:  
1. **Система коллекций** - Полностью отсутствует в backend, критическая функциональность  
2. **Флаги isFavorite/isReadLater** - Используются в UI, но не синхронизируются  
  
Эти компоненты должны быть перенесены в первую очередь для обеспечения корректной работы приложения и синхронизации данных между устройствами.
