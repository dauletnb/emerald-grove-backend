# Готовность к реализации следующего этапа

Документ фиксирует, что именно уже готово, что еще не готово, и с каким конкретным scope можно безопасно переходить к реализации в следующем сообщении.

---

## 1. Текущее состояние готовности

### Уже готово

- stage 1 по article flags уже завершен;
- frontend и backend уже согласованы по `isFavorite` и `isReadLater`;
- `POST /api/articles/sync` уже использует серверный snapshot;
- `GET /api/articles` уже возвращает флаги статей;
- collections и membership links уже имеют базовый online/offline sync-каркас;
- в проекте уже есть тестовая база для controller/service-уровня backend.

### Еще не готово полностью

- offline delete для статей не имеет полноценной sync-модели;
- offline delete для membership links не реализован;
- `collections/links sync` не возвращает подробный результат `applied/skipped`;
- нет формализованной conflict-resolution стратегии;
- AI response contract еще не полностью выровнен;
- snapshot загрузка collections пока остается N+1 схемой.

---

## 2. Что считаем зафиксированным перед следующей реализацией

Чтобы в следующем сообщении можно было сразу переходить к коду, фиксируем рабочие допущения.

### Зафиксированные архитектурные решения для ближайшей итерации

1. В ближайшей итерации **не трогаем conflict resolution**.
2. В ближайшей итерации **не меняем AI contract**.
3. В ближайшей итерации **не делаем агрегированный collections snapshot endpoint**.
4. Article sync **пока остается full-snapshot моделью**, не patch.
5. Новые изменения должны быть **additive**, без резкой поломки существующих single-item endpoints.
6. Если нужно расширять ответы sync endpoint-ов, делаем это так, чтобы frontend можно было адаптировать контролируемо в той же итерации.

### Почему именно так

Это минимизирует размер изменений и позволяет взять следующий этап как безопасный, но уже полезный кусок переноса, не расползаясь сразу на все проблемные зоны.

---

## 3. Рекомендуемый scope следующей реализации

Самый практичный следующий шаг:

## Этап 2A. Довести удаление и sync-диагностику до рабочего offline-first уровня

### Внутри этого scope

1. Добавить backend endpoints для batch sync удалений:
   - `POST /api/articles/sync/deletions`
   - `POST /api/collections/sync/deletions`
   - `POST /api/collections/links/sync/deletions`

2. Добавить result DTO для batch sync операций, как минимум для:
   - `collections/sync`
   - `collections/links/sync`
   - deletion sync endpoints

3. На frontend добавить локальные deletion queues:
   - для article external ids;
   - для collection external ids;
   - для membership links.

4. Перестроить startup/reconnect flow:
   - сначала отправить pending article updates;
   - затем pending article deletions;
   - затем collections sync;
   - затем collections deletions;
   - затем links sync;
   - затем links deletions;
   - затем сделать повторный pull server state.

### Почему это лучший следующий шаг

- он закрывает самый опасный разрыв с “воскрешением” удаленных данных;
- он делает offline-first поведение предсказуемее;
- он подготавливает основу для следующих этапов без преждевременного усложнения contract model;
- он не требует сразу вводить conflict resolution.

---

## 4. Что сознательно откладываем после следующей итерации

Чтобы не размывать реализацию, следующие задачи считаем отложенными:

- bulk sync статей отдельным endpoint-ом;
- conflict resolution через `clientUpdatedAt`;
- server-side filters для `FAVORITES`, `READ_LATER`, `RECENT`;
- выравнивание AI raw JSON contract;
- единый snapshot endpoint для collections и links.

Это важные задачи, но они не должны смешиваться с первым этапом доведения delete/sync модели.

## 5. Готовность по слоям

### Backend

Готов к следующей итерации, потому что уже есть:

- разделение controller / service / repository;
- отдельные article и collection модули;
- существующие sync endpoints, от которых можно оттолкнуться;
- тесты controller/service-уровня.

### Frontend

Тоже готов, потому что уже есть:

- IndexedDB слой с локальными статьями, коллекциями и links;
- `syncStatus`-модель;
- startup sync orchestration в `LibraryApp`;
- online/offline fallback поведение для части сущностей.

### Но есть обязательный подготовительный вывод

Frontend придется расширить не только новыми API вызовами, но и новой локальной очередью удалений. Текущей схемы `delete locally -> try DELETE -> hope for success` уже недостаточно.

## 7. Что нужно реализовывать в каком порядке

### Порядок работ внутри следующей итерации

1. Сначала ввести backend DTO для deletion sync и result payload.
2. Затем реализовать backend service/controller endpoints.
3. Затем покрыть это backend тестами.
4. После этого расширить frontend IndexedDB schema и очереди удалений.
5. Затем обновить frontend API clients.
6. Затем изменить orchestration в `LibraryApp`.
7. В конце проверить сценарии:
   - online delete;
   - offline delete;
   - reconnect sync;
   - skip/partial result handling.

### Почему порядок именно такой

Frontend delete flow нельзя надежно собирать, пока не зафиксирован backend contract для deletion sync.

---

## 8. Minimal acceptance criteria для следующей реализации

Следующая итерация считается завершенной, если выполняется все ниже:

1. Удаленная offline статья не возвращается после reconnect.
2. Удаленная offline коллекция не возвращается после reconnect.
3. Удаленная offline membership link не возвращается после reconnect.
4. Batch sync deletions работает идемпотентно.
5. Frontend после sync делает повторный pull server state.
6. При skip или partial apply frontend получает понятный результат, а не только `200 OK`.

---

## 9. Что можно считать готовностью “можно начинать”

На текущий момент проект **готов к началу следующей реализации**, если мы принимаем следующий рабочий пакет:

- цель итерации: deletion sync + sync diagnostics;
- article sync остается full-snapshot;
- conflict resolution не входит в эту итерацию;
- AI contract не входит в эту итерацию;
- collections snapshot optimization не входит в эту итерацию.

Это достаточно узкий и реалистичный scope, чтобы переходить к коду сразу в следующем сообщении.

---

## 10. Команда для следующего шага

Если идти по рекомендованному пути, следующий запрос на реализацию можно формулировать так:

> Начинай реализацию этапа 2A: deletion sync + sync diagnostics для статей, коллекций и membership links на backend и frontend.

Именно после такой постановки уже можно без дополнительного анализа переходить к изменениям в коде.
