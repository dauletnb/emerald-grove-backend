# Критические моменты для следующего переноса после stage 1

Документ фиксирует критические моменты для дальнейшего переноса логики из Frontend в Backend с учетом уже завершенного первого этапа, описанного в `stage_1_total.md`.

---

## 1. Удаления остаются самым опасным незавершенным местом

Самый рискованный разрыв сейчас находится в удалениях:

- offline-delete для статей фактически не доведен до backend-модели;
- offline-delete для membership-связей вообще не поддержан полноценно.

### Что происходит сейчас

Во frontend статья сначала удаляется локально, а затем отправляется `DELETE` на сервер.

Если сеть недоступна:

- локальная запись уже удалена;
- backend об этом не узнает;
- tombstone или очередь удаления не сохраняются;
- при следующем `GET /api/articles` статья может снова появиться.

Для membership-связей ситуация еще жестче:

- offline remove в UI прямо не поддержан;
- пользователь получает сообщение, что offline removal пока не поддерживается.

### Почему это критично

После этапа 1 backend уже начинает выступать как source of truth для состояния статьи. Но удаление все еще живет по фронтендовой схеме и не встроено в общую модель синхронизации.

### Что нужно учитывать при следующем переносе

Нужно выбрать и реализовать одну из полноценных моделей:

- tombstone-записи для статей и links;
- отдельную очередь pending delete-операций;
- bulk delete sync endpoint-ы;
- либо серверную reconciliation-модель, где удаление тоже является синхронизируемым событием.

Без этого перенос логики в backend будет неполным и приведет к "воскрешению" данных после повторного pull.

---

## 2. Sync коллекций и links пока только добавляет, но не удаляет

Текущий sync для коллекций и membership links несимметричен:

- коллекции можно синхронизировать bulk-ом;
- links можно синхронизировать bulk-ом;
- но удаление links и удаление коллекций в offline-sync модели не доведены до полноценного server contract.

### Что происходит сейчас

Frontend:

- формирует pending payload для коллекций;
- формирует pending payload для links;
- после sync полностью перезагружает collections state с сервера.

Backend:

- в `syncCollectionLinks` просто пропускает записи, если не найдена статья или коллекция;
- делает это через `continue`;
- пишет warning в лог, но не возвращает клиенту структуру skipped-элементов.

### Почему это критично

Из-за этого возможен silent data loss сценарий:

1. пользователь локально создал или изменил связи;
2. при sync часть данных была пропущена;
3. frontend затем сделал полный `replaceCollectionsState`;
4. локальное намерение пользователя пропало без явной ошибки в UI.

### Что нужно учитывать при следующем переносе

Следующий этап переноса должен закрыть хотя бы один из вариантов:

- backend возвращает отчет по `applied / skipped / conflicted`;
- frontend хранит локальные tombstone/queue до подтвержденного применения;
- появляется полноценный server-side snapshot endpoint с информацией о конфликтах;
- либо вводится version/conflict model для collections и links.

---

## 3. Article sync уже серверный, но он full-snapshot, а не patch

Это не ошибка, но это критичное архитектурное ограничение.

### Что уже сделано правильно

После этапа 1 frontend уже:

- отправляет `isFavorite`;
- отправляет `isReadLater`;
- читает `response.article`, а не `response.payload`;
- сохраняет серверный snapshot обратно в локальный store.

### В чем риск

`POST /api/articles/sync` сейчас работает как полная синхронизация снимка статьи.

Это значит:

- boolean-поля нельзя "не отправить";
- notes нельзя частично игнорировать, если endpoint считается авторитетной синхронизацией статьи;
- любое дальнейшее расширение article-модели увеличит риск случайного затирания данных при неполном payload.

### Почему это критично для следующих этапов

Если на backend будут переноситься новые article-поля, frontend больше не сможет безопасно жить на полусобранных DTO или на частичных обновлениях "по месту".

### Что нужно учитывать при следующем переносе

Нужно заранее определить стратегию:

- либо сохранить full-snapshot модель и строго поддерживать полную сборку payload;
- либо ввести отдельные patch endpoint-ы для точечных операций;
- либо разделить синхронизацию статьи на отдельные bounded contract-ы: flags, notes, metadata, AI state.

Иначе с каждым новым перенесенным полем цена ошибки в sync будет расти.

---

## 4. Есть скрытое контрактное расхождение по AI-результату

Это отдельная зона риска, потому что она может ломаться не сразу и неочевидно.

### Что ожидает frontend

Frontend работает так, как будто `GET /api/articles/{externalId}/ai` возвращает:

- `aiStatus`
- `aiResult` как уже готовый объект структуры `AIResult`

### Что отдает backend

Backend возвращает:

- `aiStatus`
- `aiResult` как raw JSON string

### Почему это критично

Пока это может выглядеть рабочим только при неявном совпадении формата. Но по текущему frontend-коду нет надежной явной адаптации этого ответа в типизированный объект.

При дальнейшем переносе AI-логики в backend это место станет точкой нестабильности:

- UI может сломаться на parse/runtime-уровне;
- типы frontend и backend будут расходиться;
- ошибка может проявляться только на части статей.

### Что нужно учитывать при следующем переносе

Нужно унифицировать контракт:

- либо backend возвращает уже нормализованный JSON-объект;
- либо frontend явно парсит raw JSON string и валидирует структуру;
- лучше закрепить это тестами контрактного уровня.

---

## 5. Snapshot коллекций сейчас масштабируется плохо

Текущая схема загрузки collections работает, но архитектурно остается временной.

### Что происходит сейчас

Frontend делает:

1. `GET /api/collections`
2. затем для каждой коллекции отдельный `GET /api/collections/{id}/articles`

После этого локально пересобирает membership snapshot.

### Почему это критично

При росте числа коллекций:

- увеличивается число запросов;
- удлиняется старт библиотеки;
- возрастает вероятность частичного сбоя на одном из запросов;
- перенос source of truth в backend делает эту зависимость еще сильнее.

### Что нужно учитывать при следующем переносе

На следующем этапе лучше перейти к одному из вариантов:

- единый snapshot endpoint для collections и links;
- агрегированный endpoint по всем membership-связям;
- server response, который уже содержит collections вместе с article ids.

Иначе дальнейший перенос в backend будет усиливать сетевую связанность вместо упрощения архитектуры.

## 6. Локальная дедупликация статьи все еще завязана на `id || url`

Сейчас это разумный компромисс, но он остается чувствительным местом.

### Что происходит сейчас

В IndexedDB-слое статья ищется по одному из признаков:

- совпадение `id`;
- или совпадение `url`.

Это согласуется с тем, что backend умеет находить существующую статью по:

- `externalId`;
- либо по `url + userId`.

### Почему это критично

Если правила дедупликации на backend будут меняться дальше, а frontend останется на старой логике:

- статьи могут начать склеиваться по разным правилам;
- или, наоборот, дублироваться локально и на сервере по-разному;
- появятся трудноотлавливаемые расхождения между IndexedDB и backend.

### Что нужно учитывать при следующем переносе

Нужно закрепить дедупликацию как явный контракт:

- по каким полям backend считает статью той же самой;
- что делать при изменении URL;
- что делать при пустом `externalId`;
- что считается первичным ключом в offline-first режиме.

---

## Что уже хорошо совпадает с итогами stage 1

На текущий момент уже корректно встроены важные результаты первого этапа:

- frontend хранит `isFavorite` и `isReadLater` в основной article-модели;
- `POST /api/articles/sync` отправляет эти поля;
- frontend читает `response.article`;
- `GET /api/articles` используется как серверный источник актуального article state;
- при старте библиотеки сначала синхронизируются статьи, затем коллекции и links.

Это означает, что перенос article flags в backend в целом уже встроен правильно.

---

## Главный вывод

После первого этапа основные критические моменты сместились:

- уже не в типах;
- уже не в простом добавлении полей;
- а в семантике синхронизации и конфликтов.

Следующие этапы переноса нужно строить вокруг пяти ключевых тем:

1. удаление как синхронизируемое событие;
2. симметричный sync для collections и links;
3. явная стратегия full-snapshot vs patch;
4. выравнивание AI-контракта;
5. агрегированный server snapshot вместо N+1 загрузки.

Без этого backend будет формально хранить больше данных, но не станет полностью надежным центром истины для offline-first сценария.
